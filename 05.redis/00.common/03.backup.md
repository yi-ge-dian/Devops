1. redis 单机如何迁移到集群

```shell
redis-cli -a 123456 --cluster import 10.0.0.181:6379 --cluster-from 10.0.0.180:6379 --cluster-from-pass 123456 --cluster-replace

2. redis 集群如何备份数据
```shell
redis-cli -a 123456 --cluster backup 10.0.0.181:6379 /data/6379/backup
# 或者
# 登录到各个 master 或者 slave，推荐slave，执行 bgsave
```
3. 恢复数据

3.1. 如果新节点想同时开启 rdb和aof两个模式，但只有旧节点的rbd文件，如何恢复


```text
执行后会生成 dump.rdb 文件, 如果开启了 AOF，会生成 appendonly.aof 目录。
Redis运行中
    ├── 新的写命令 → 实时写入 appendonly-6379.aof.1.incr.aof (增量日志)
    │
    ├── 当满足重写条件时 (AOF文件翻倍或达到1GB)
    │   └── 自动触发 BGREWRITEAOF
    │       └── 生成新的 appendonly-6379.aof.1.base.rdb (全量快照)
    │           └── 清空 .incr.aof 重新记录新命令
    │
    └── 当你手动执行 BGSAVE
        └── 生成独立的 dump.rdb (独立的备份文件)

不要删除 dump.rdb：它虽然和AOF的功能重叠，但是手动备份和迁移的最佳选择
不要手动删除AOF相关文件：Redis依赖它们进行正常的数据持久化和重启恢复
备份策略：
- 日常运维备份：使用 BGSAVE 生成的 dump.rdb，因为文件独立易管理
- 高可用保障：依赖AOF机制，Redis重启时会自动使用 .base.rdb + .incr.aof 完整恢复

appendonly-6379.aof.1.base.rdb 是Redis自己用来保证数据安全的内部文件，而 dump.rdb 是你用来做外部备份的管理文件。两者共存但不冲突，各司其职。

# 步骤1: 新节点临时关闭AOF
# 修改新节点的 redis.conf
appendonly no

# 步骤2: 停止新节点Redis
redis-cli -a 123456 -h <new-ip> -p <new-port> SHUTDOWN NOSAVE

# 步骤3: 替换dump.rdb文件
cp /backup/old-dump.rdb /data/6379/data/dump.rdb
chown redis:redis /data/6379/data/dump.rdb

# 步骤4: 启动新节点（此时AOF未开启，会从dump.rdb加载）
redis-server /path/to/redis.conf

# 步骤5: 验证数据已恢复
redis-cli -a 123456 -h <new-ip> -p <new-port> DBSIZE
redis-cli -a 123456 -h <new-ip> -p <new-port> GET <test-key>

# 步骤6: 在线开启AOF（无需重启）
redis-cli -a 123456 -h <new-ip> -p <new-port> CONFIG SET appendonly yes

# 步骤7: 触发AOF重写，生成完整的AOF文件
redis-cli -a 123456 -h <new-ip> -p <new-port> BGREWRITEAOF

# 步骤8: 确认AOF文件生成
ls -lh /data/6379/data/appendonlydir/
# 应该会看到类似文件：
# appendonly-6379.aof.1.base.rdb
# appendonly-6379.aof.1.incr.aof
# appendonly-6379.aof.manifest

# 步骤9: (可选)持久化配置到redis.conf
redis-cli -a 123456 -h <new-ip> -p <new-port> CONFIG REWRITE
````

3.2. 如果新节点想同时开启 rdb和aof两个模式，但有旧节点的rbd文件和aof文件，如何恢复，直接复制过去，启动即可，rbd 文件无需拷贝。
